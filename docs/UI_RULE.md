# UI_RULE

営業手帳のUIは、スマホでは現場入力、PCでは業務管理に最適化する。

## 基本方針

- 既存機能を壊さない。
- スマホは5タブ構成を維持する。
- PCは左サイドバーとテーブル中心の全画面UIを維持する。
- 入力フォームは押しやすく、読みやすく、誤操作しにくくする。
- 重要情報はカード、ステータス、バッジ、テーブル列で素早く確認できるようにする。

## スマホUI

- 767px以下をスマホUIとする。
- 下部ナビは「ホーム」「取引先」「＋追加」「案件」「商品」を基本とする。
- カード型UIを基本とする。
- 入力欄は縦並びにする。
- 主要ボタンは親指で押しやすい位置とサイズにする。
- 横スクロールを発生させない。

## PC UI

- 768px以上をPC UIとする。
- 下部ナビは非表示にし、左サイドバーを表示する。
- 画面幅を最大限使う。
- 一覧はテーブル中心にする。
- 詳細画面は2〜3カラムを基本とする。
- 検索・フィルターは上部固定を基本とする。
- 入力フォームは2列レイアウトを基本とし、長文欄は全幅にする。

## 在庫UI

在庫ステータスは色付きバッジで表示する。

- 未連携: グレー
- 在庫あり: グリーン
- 残少: オレンジ
- 欠品: レッド
- 入荷待ち: ブルー
- 取扱停止: ダークグレー
- 要確認: パープル

表示方針:

- 商品一覧では在庫ステータスを必ず表示する。
- 商品詳細では現在庫、利用可能数、ロット、賞味期限、入荷予定を表示する。
- 欠品・残少・賞味期限間近は目立つ警告にする。
- スマホでは在庫情報をカード内にまとめる。
- PCでは商品テーブルの列として表示する。

## 見積PDF UI

- 見積詳細に「PDF作成」「PDF確認」「PDFダウンロード」導線を配置する。
- PDF作成前に見積内容を確認できるようにする。
- 自動送信ボタンは置かない。
- PDF生成中はローディングを表示する。
- 生成失敗時はエラーを表示し、見積データは壊さない。
- 版数、生成日時、生成者を見えるようにする。
- 確定済みPDFは上書きせず、履歴として表示する。

## 経営判断ダッシュボードUI

- 重要指標は大きな数字カードで表示する。
- PCでは複数カードを横並びにする。
- スマホでは縦並びカードにする。
- グラフ、ランキング、期限切れリストを分けて表示する。
- フィルターは期間、担当者、商品、顧客ランク、ステータスを基本とする。
- ダッシュボードから該当する顧客カルテ、商品詳細、見積詳細へ遷移できるようにする。

## 表示速度

- 一覧画面では画像・添付ファイル本体を読み込まない。
- StorageファイルはURLとメタ情報のみ一覧表示する。
- 重い集計は `dashboardService` に分離する。
- 顧客カルテや分析画面は必要時のみ読み込む。
- 検索結果は可能な範囲でメモ化する。

## 入力フォーム

- `input`、`textarea`、`select` はPC・スマホとも入力可能であることを最優先する。
- 不要な `readOnly`、`disabled`、`pointer-events: none` を使わない。
- ラベルと入力欄の対応を明確にする。
- 数値入力は保存時に数値として扱い、表示時に整形する。
- ファイル添付はStorage保存を前提とし、LocalStorageへ本体を保存しない。
